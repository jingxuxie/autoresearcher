\begin{abstract}
We present the \intern, a repository-centered pipeline for human-guided scientific prototyping with language-model agents. Rather than treating current agents as fully autonomous scientists, the \intern\ decomposes research into bounded planning, execution, review, summarization, and checkpoint-supervision roles. It stores plans, commands, metrics, artifacts, reviews, and claim boundaries as auditable repository state, and escalates high-stakes decisions to a \gptpro\ checkpoint supervisor. Across two live reinforcement-learning case studies, the system produced scoped positive evidence, decision-relevant negative results, and stopping decisions that prevented unsupported scaling. We argue that this research-intern paradigm is a practical human--AI collaboration model for fast, large-scale prototyping of scientific ideas.
\end{abstract}

\section{Introduction}

Scientific work is often described through breakthroughs: a new idea, a new theorem, a surprising experimental result, or a new algorithm. In daily computational research, however, progress often depends on a more mundane and expensive loop: formulate a hypothesis, translate it into code, build a minimal diagnostic, choose a fair baseline, run the experiment, inspect artifacts, update the hypothesis, and write down what happened. For machine-learning researchers, much of this loop is not blocked by creativity alone. It is blocked by engineering bandwidth. A scientist may have many plausible ideas, but each idea requires environment setup, small benchmark construction, scripts, metrics, baselines, validation commands, plots, result interpretation, and careful documentation. Many ideas should be stopped early, but weak evidence is easy to miss when it lives in scattered logs and unstructured model summaries.

This bottleneck motivates language-model systems for scientific discovery. Recent systems branded as ``AI scientists,'' ``autonomous researchers,'' or ``autoresearchers'' attempt to automate increasingly large portions of the research process, from idea generation to coding, experimentation, and manuscript writing~\citep{lu2024aiscientist,yamada2025aiscientistv2,schmidgall2025agentlab,ghareeb2025robin,ghafarollahi2024sciagents}. These systems are important: they demonstrate that language models can coordinate tools, modify repositories, execute experiments, and produce research-like artifacts. At the same time, we believe the phrase ``autoresearcher'' is currently aspirational. Today's systems are not yet reliable end-to-end scientists. Their outputs still require human framing, expert review, artifact checking, baseline skepticism, and claim control. Benchmarks of machine-learning and software-engineering agents report substantial variability, long-horizon planning failures, hallucination risks, and dependence on environment design~\citep{huang2023mlagentbench,jimenez2023swebench,yang2024sweagent}. Agent Laboratory explicitly studies human feedback during stages of an automated research pipeline, underscoring that human guidance remains valuable even when agents can run substantial workflows~\citep{schmidgall2025agentlab}.

The question, then, is not only whether we can build a fully autonomous scientist. It is also what kind of partial autonomy is most useful to human researchers today. We argue that many scientists need an \intern: a system that behaves like a diligent research intern rather than an independent principal investigator. The human scientist supplies research taste, high-level hypotheses, scope, and publication judgment. The AI intern turns these ideas into small falsifiable prototypes, runs controlled diagnostics, records negative results, maintains an auditable lab notebook, and asks for senior guidance when a decision becomes ambiguous. This collaboration pattern is valuable even if full autonomy remains out of reach. It can make fast and large-scale prototyping possible: instead of manually testing one idea at a time, a scientist can maintain a portfolio of ideas, each advanced by bounded experiments and stopped when evidence is weak.

This paper presents a prototype of that paradigm: the \intern. The system is a repository-centered orchestration pipeline for human--AI scientific prototyping. A project begins with a human-written charter specifying the research goal, main hypothesis, scope, metrics, success criteria, failure criteria, and compute budget. A local supervisor agent proposes exactly one next experiment or chooses to pivot, stop, request human review, or request a \gptpro\ checkpoint. An executor agent implements and runs the experiment under a hard wall-clock budget. A reviewer agent audits the result and states whether automatic continuation is allowed. A summarizer agent writes progress reports and claim boundaries. A deterministic orchestrator validates every structured artifact, commits project state, enforces stop conditions, and escalates high-stakes decisions to a \gptpro\ checkpoint supervisor.

The central design principle is \emph{evidence-gated co-autonomy}. The \intern\ allows agents to propose and execute experiments, but not to silently convert activity into scientific progress. Claims must be grounded in machine-readable result files, declared artifacts, baseline metrics, reviewer verdicts, and metric ledgers. Continuation is allowed only when both deterministic validators and model-based reviewers accept the evidence. Pivots, stop decisions, ambiguous reviews, timeouts, repeated weak passes, protected-file drift, and publication-oriented claims can trigger a stronger \gptpro\ checkpoint or human review. The repository, rather than any chat transcript, is the authoritative memory.

This design changes the collaboration pattern between a scientist and an AI system. Instead of asking a coding assistant to write isolated code snippets, the scientist delegates an entire bounded research loop. Instead of trusting an agent's prose summary, the scientist receives a structured trail of plans, commands, metrics, artifacts, reviews, and decisions. Instead of deciding manually after every run, the scientist specifies when the system may continue locally and when it must escalate. The human focuses on taste, framing, and claim boundaries; the agent handles implementation, bookkeeping, and preliminary falsification.

We evaluate the \intern\ through two live reinforcement-learning research projects. The first, which we call \textbf{Soft Reward-to-Goal Conversion}, tests whether soft successor-measure terminal marginalization can preserve reward-to-goal equivalence while reducing the variance of sampled terminal success events. In this project, the \intern\ progressed from one-state variance diagnostics to tabular equivalence checks, RiverSwim coverage tests, FourRooms vector-goal checks, and a low-rank auxiliary-goal negative result. It produced a scoped internal write-up: soft terminal marginalization is supported as a small-tabular variance-reduction and normalized-Q-equivalence mechanism under adequate coverage, while the tested rank-4 low-rank auxiliary-goal approach is unsupported and showed negative transfer. The second, which we call \textbf{Stochastic Long-Horizon Goal Reaching}, tests stochastic extensions of Transitive RL (TRL). This project produced useful diagnostics, but ultimately stopped after showing that apparent stochastic-TRL improvements were explained by fairer baselines or representation/context sufficiency rather than a distinct algorithmic advantage. We use Soft Reward-to-Goal Conversion as the main case study and summarize Stochastic Long-Horizon Goal Reaching in the appendix.

This paper makes four contributions. First, we introduce evidence-gated co-autonomy, a human--AI collaboration pattern in which agents can run bounded research loops but continuation depends on structured evidence and explicit gates. Second, we describe a concrete architecture for repository-centered scientific memory, including structured plans, results, reviews, metric ledgers, progress summaries, and checkpoint packets. Third, we show how a stronger supervisor model, \gptpro, can be used selectively as a senior checkpoint reviewer rather than as the default executor. Fourth, we report two live case studies showing that the system can produce both positive and negative scientific evidence, including stopping decisions that prevent unsupported scaling.

\section{Related Work}

\subsection{Autonomous AI scientists and scientific agents}

The most direct comparison is with systems that attempt broad automation of scientific discovery. The AI Scientist proposes an end-to-end pipeline for idea generation, experiment execution, manuscript writing, and automated review~\citep{lu2024aiscientist}. AI Scientist-v2 extends this direction with agentic tree search and workshop-level paper generation~\citep{yamada2025aiscientistv2}. Agent Laboratory studies LLM agents as research assistants across literature review, experimentation, and report writing, while allowing users to provide feedback~\citep{schmidgall2025agentlab}. Robin explores a multi-agent system for automating scientific discovery in biology~\citep{ghareeb2025robin}; SciAgents uses multi-agent graph reasoning for scientific discovery~\citep{ghafarollahi2024sciagents}. Surveys have begun to organize this area as LLM-based scientific agents or as movement from automation toward autonomy in scientific workflows~\citep{ren2025scientificagents,zheng2025automationautonomy,zhang2025scientificmethod}.

Our work differs in emphasis. We do not claim that the system is a complete autonomous scientist. Instead, we design for the intermediate regime that many researchers actually need: a system that can carry out intern-level prototyping while preserving human authority over framing, pivots, scale-up, and publication. This makes the main technical object not only an agent, but a workflow contract between a human scientist, a repository, and a set of bounded model roles.

\subsection{AI for scientific discovery beyond language agents}

AI systems have already contributed to major scientific and mathematical advances, including protein-structure prediction with AlphaFold~\citep{jumper2021alphafold}, algorithm discovery with AlphaTensor~\citep{fawzi2022alphatensor}, mathematical program search with FunSearch~\citep{romeraparedes2024funsearch}, and materials discovery with graph neural networks~\citep{merchant2023gnome}. These systems typically combine domain-specific representations, search, simulation, or learning objectives with strong evaluation procedures.

The \intern\ is complementary. It is not a domain-specific discovery engine. It is a workflow scaffold for early prototyping: it helps a human scientist instantiate a hypothesis, design small diagnostic experiments, run them, and decide what the evidence supports. In that sense, it is closer to a programmable lab notebook and research assistant than to a single scientific discovery algorithm.

\subsection{Language-agent scaffolding, tool use, and reflection}

The \intern\ builds on general language-agent scaffolding. ReAct interleaves reasoning and acting so that language models can use tools and update plans from observations~\citep{yao2022react}. Toolformer studies models that decide when and how to call external tools~\citep{schick2023toolformer}. Tree of Thoughts explores search over intermediate reasoning states~\citep{yao2023tree}. Reflexion shows how agents can improve by storing verbal reflections over prior trials~\citep{shinn2023reflexion}. Voyager combines an automatic curriculum, executable skill library, and iterative prompting for open-ended embodied learning~\citep{wang2023voyager}.

Our contribution is not a new inference-time reasoning algorithm. Instead, we adapt these ideas to scientific workflow design. The \intern\ has explicit roles: supervisor, executor, reviewer, summarizer, and checkpoint supervisor. It uses repository artifacts as long-term memory, rather than relying only on hidden chat context or a reflection buffer. It also distinguishes reflection from evidence. A reviewer may reflect on whether an experiment supports a claim, but the orchestrator requires artifacts and schemas before accepting that reflection. This separation is crucial in science, where persuasive explanations are not substitutes for reproducible data.

\subsection{Coding agents, benchmarks, and reproducible experimentation}

Scientific prototyping often looks like software engineering: the agent must navigate a repository, edit code, run commands, and interpret failures. SWE-bench evaluates language models on real GitHub issues~\citep{jimenez2023swebench}. SWE-agent argues that agent-computer interface design is central to automated software engineering performance~\citep{yang2024sweagent}. MLAgentBench evaluates language agents on machine-learning experimentation and identifies challenges such as long-term planning and hallucination~\citep{huang2023mlagentbench}. Work on SWE-Bench+ further highlights that apparent benchmark success can be affected by leakage and weak tests~\citep{aleithan2024swebenchplus}. These findings motivate the \intern's deterministic validators, artifact checks, and protected-file drift gates.

Our setting differs from code repair benchmarks. A scientific prototype is not solved by passing unit tests alone. The system must decide whether the experiment tested the right hypothesis, whether the baseline is fair, whether a negative result is informative, and whether a claim should be narrowed. This is why the \intern\ stores not only code diffs and test outputs but also plans, result schemas, baseline metrics, reviewer verdicts, metric ledgers, progress summaries, and checkpoint packets.

\subsection{Reinforcement-learning background for the case studies}

Both case studies involve goal-conditioned reinforcement learning (GCRL), a setting in which agents condition behavior on goals rather than a single task reward~\citep{liu2022gcrlsurvey}. Hindsight Experience Replay is an early influential method for using achieved goals as additional supervision~\citep{andrychowicz2017her}. Offline GCRL benchmarks such as OGBench emphasize long-horizon reasoning, stochasticity, and stitching challenges~\citep{park2024ogbench}. Transitive RL proposes a divide-and-conquer value-learning rule for offline GCRL~\citep{park2025trl}. Classical Q-learning and dynamic programming provide reference baselines for the tabular diagnostics used in our projects~\citep{watkins1992qlearning,sutton2018rl}. These references are not the contribution of this paper, but they explain why small exact-DP diagnostics are scientifically useful: they expose whether an agentic pipeline is producing real algorithmic evidence or only artifacts of coverage, scaling, or baseline choice.

\subsection{Design goals}

The \intern\ is designed for early-stage scientific prototyping. It is not a benchmark optimizer, an autonomous paper factory, or a replacement for expert review. Its goals are: (1) translate rough ideas into falsifiable small experiments; (2) maintain an auditable record of plans, commands, metrics, and evidence; (3) run quickly enough that many ideas can be screened; (4) stop weak directions before they consume large compute; and (5) help the human scientist preserve claim boundaries. These goals lead to four design choices.

First, the system is \emph{bounded}. Experiments are intentionally small. The default executor timeout is 30 minutes, and the charter can prohibit large training runs, large downloads, expensive API calls, or benchmark-scale experiments without approval. This makes the pipeline useful for fast and large-scale prototyping of ideas: a scientist can test many hypotheses at low cost and reserve expensive validation for ideas that survive early falsification.

Second, the system is \emph{role-specialized}. Planning, execution, review, summarization, and senior checkpointing are separated. This prevents a single agent from both producing and uncritically accepting its own result. It also lets different roles operate under different permissions: the executor can modify experiment code, while the reviewer and supervisor normally run read-only.

Third, the system is \emph{repository-centered}. Scientific memory lives in versioned files, not in a chat transcript. Role-resumed agent sessions provide continuity, but every role is instructed that repository files are the source of truth. This makes it possible to audit the project after many iterations, reset or replace an agent, and still recover the research state.

Fourth, the system is \emph{evidence-gated}. Every transition is mediated by validators or reviews. A result that omits artifact paths, fails schema validation, or lacks a reviewer verdict cannot silently advance the project. A stop or pivot decision can trigger \gptpro\ checkpoint supervision. A draft intended for external use must include red-line unsupported claims and a pre-publication review checklist.

\subsection{Human charter as the root of research intent}

Every project begins with a human-written charter. The charter is intentionally short but semantically important. It specifies the research goal, main hypothesis, initial scope, primary metrics, success criteria, failure criteria, runtime budget, and first-experiment guidance. This document is the contract between the human scientist and the agent system.

The charter serves several functions. It prevents the local supervisor from drifting toward unrelated experiments. It defines what early evidence should count. It states what would disprove or block the idea. It encodes compute limits and red lines. It also makes human taste explicit: the scientist can say which baselines matter, which settings are too expensive, and which results would be scientifically meaningful. In our case studies, charters prevented premature moves to large RL benchmarks and forced the pipeline to begin with exact-DP tabular diagnostics.

\subsection{Role-specialized agents}

The \intern\ uses five roles. The \emph{supervisor} reads the charter, state, metric ledger, latest results, and latest reviews. It decides whether to continue, pivot, stop, request human input, or request a \gptpro\ review. If it continues, it proposes exactly one small experiment with success criteria, failure criteria, required outputs, and estimated runtime. The motivation is to make planning falsifiable. A vague instruction such as ``try more variants'' is not acceptable; the next experiment must test a named claim.

The \emph{executor} receives the plan and runs the experiment. It may edit code and create artifacts, but it must write a structured result file and a human-readable summary. It records exact commands, metrics, baseline metrics, artifacts, interpretation, known failures, and next questions. The executor is not allowed to change the research goal, delete prior results, or run large experiments outside the budget. If it cannot complete the plan, it writes a failed, blocked, or timeout result rather than leaving the project ambiguous.

The \emph{reviewer} audits the result after execution. It checks whether the required files exist, whether the artifact paths are valid, whether the baseline is fair, whether success criteria were satisfied, whether failure criteria were triggered, and whether the interpretation overclaims the evidence. It returns a verdict and a boolean indicating whether automatic continuation is allowed. The reviewer exists because executor summaries are claims, not facts.

The \emph{summarizer} periodically writes progress reports. These reports are not only for convenience. They compress the research state into a human-readable lab notebook: current status, experiment ledger, main findings, limitations, recommended human decision, and files to inspect. Summaries also become inputs to \gptpro\ checkpoint packets.

Finally, the \emph{checkpoint supervisor} is a stronger model, \gptpro, invoked only at important gates. It is not used for every routine action. Instead, it reviews compact packets when the local loop proposes a pivot or stop, accumulates weak passes, reaches a cadence checkpoint, times out, encounters protected-file drift, or approaches a publishable claim. This senior-review pattern uses strong reasoning where it is most valuable without making the entire loop expensive or brittle.

\subsection{Repository-centered memory and data formats}

The repository is organized as a lab notebook with machine-readable structure. Each project contains a charter, state file, plans, results, reviews, decisions, progress summaries, metric ledgers, \gptpro\ checkpoint packets, and artifacts. The state file tracks iteration count, status, failure streak, no-progress rounds, last decision, last checkpoint, protected-file drift, and whether human review is required. Plans are stored as both JSON and Markdown. Results are stored as JSON plus human summaries. Reviews are stored as JSON plus Markdown. Metric ledgers summarize decision-relevant quantities across iterations.

The result schema is central. A result must name the experiment id, status, claim tested, commands run, metrics, baseline metrics, artifact paths, interpretation, known failures, and next questions. This schema forces the executor to connect actions to evidence. The review schema forces the reviewer to state a verdict, evidence checked, reasons, required fixes, and risk flags. The \gptpro\ decision schema forces \gptpro\ to choose continue, pivot, stop, or needs-human and to provide evidence, risks, and a next experiment only when appropriate.

This memory design solves a practical problem with long-running agent conversations. Role-resume sessions and model context can be useful, but they are not trustworthy project databases. They may omit details, summarize incorrectly, or drift. Repository memory is versioned, inspectable, and portable across agents. A future model or human can reconstruct the project from files alone.

\subsection{Grounding claims in evidence}

Scientific overclaiming is a major failure mode for agentic workflows. Goodhart's law warns that proxy measures can become misleading targets when optimized directly~\citep{goodhart1975problems}; in agentic science, the proxy may be activity, a plausible summary, or a single metric detached from its baseline. The \intern\ addresses this by requiring every claim to be linked to result artifacts. Each experiment begins with success and failure criteria. Each result records the claim tested and baseline metrics. The reviewer checks whether the interpretation follows from the metrics. The metric ledger extracts important scalars, positive signals, and negative signals across iterations. The progress summary separates confirmed findings, limitations, and recommended human decisions.

This structure creates a claim-to-evidence map. In the Soft Reward-to-Goal Conversion case, the final internal draft distinguishes supported estimator claims from unsupported auxiliary-representation claims. In the Stochastic Long-Horizon Goal Reaching case, the final checkpoint decision stops the project because fair model-DP baselines explain apparent gains. These outcomes are possible because the system does not treat a successful run as scientific success. It asks: what claim did the experiment test, against what baseline, with what artifact-backed evidence?

\subsection{Reflection, self-checking, and checkpoint escalation}

The \intern\ uses reflection in a constrained way. Agents do not merely write free-form thoughts. The reviewer reflects on the executor's claims by checking artifacts and baselines. The summarizer reflects on the project trajectory by compressing evidence into status and limitations. The supervisor reflects on whether continuation is justified. \gptpro\ reflects on local decisions at high-stakes checkpoints.

The system also distinguishes between recoverable failures and scientific failures. Invalid JSON, missing artifacts, timeout, and bridge failures are operational failures; they are recorded and may be retried. Negative metrics, baseline equivalence, and triggered failure criteria are scientific evidence; they may justify stopping. This distinction prevents the loop from confusing infrastructure problems with research conclusions.

A key design choice is that reflection is always externalized. The reviewer writes a review file. The summarizer writes a progress report. The checkpoint supervisor writes a decision file. The orchestrator commits these outputs. This turns agent reflection into inspectable evidence for the human scientist rather than hidden private reasoning.

\subsection{Autonomy boundaries and safety gates}

The \intern\ is deliberately conservative. Each executor run has a wall-clock timeout. Artifacts must exist. Result JSON must satisfy a schema. Artifact paths must remain inside the repository. Role permissions are separated, so planning and review agents do not normally edit source code. Protected-file drift triggers a checkpoint. Stop, pivot, timeout, repeated weak pass, and publication-oriented claims can trigger \gptpro\ or human review.

These gates are important because scientific agents can otherwise reward activity. It is easy for an agent to propose another variant, run another sweep, or generate another plausible explanation. The \intern\ is designed to make that behavior harder. It is easier to continue when the next experiment is cheap and information-rich; it is harder to move to OGBench, AntMaze, or neural training before tabular diagnostics pass, to claim a baseline win without recording raw metrics, or to transform a narrow negative result into a broad conclusion. The system's default behavior is conservative: when evidence is ambiguous, stop, escalate, or ask for a human decision.

\section{Evaluation Protocol}

We evaluate the \intern\ through live use on two research projects proposed by the human scientist. This is not a randomized productivity study; we do not yet measure human time saved against a manual baseline. Instead, we ask whether the pipeline exhibits behaviors required of a useful research intern:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Translation.} Can the system convert a rough idea and charter into concrete experiments?
    \item \textbf{Execution.} Can it implement and run small diagnostics with required artifacts?
    \item \textbf{Grounding.} Does it tie claims to metrics, commands, baseline comparisons, and evidence files?
    \item \textbf{Epistemic discipline.} Does it preserve negative evidence and avoid unsupported scaling?
    \item \textbf{Autonomy control.} Does it know when to continue locally and when to escalate, stop, or ask for human guidance?
\end{enumerate}

The two projects are both reinforcement-learning-style computational ideas, but they differ in outcome. Soft Reward-to-Goal Conversion produced scoped positive estimator evidence and scoped negative auxiliary evidence, culminating in an internal short-paper draft. Stochastic Long-Horizon Goal Reaching produced useful diagnostics but ultimately stopped after stronger baselines explained apparent algorithmic gains. This contrast is useful: the \intern\ should not be judged only by whether it finds positive results. It should also be judged by whether it can narrow claims and terminate weak directions.

\section{Main Case Study: Soft Reward-to-Goal Conversion}

\subsection{Motivation and starting idea}

The first project began from a simple but practically important question in goal-conditioned reinforcement learning. Suppose a researcher wants to transform an ordinary reward-maximization problem into a goal-reaching problem by adding an artificial ``success'' goal. A sampled conversion can create sparse terminal success events: the learner sees rare samples indicating that the artificial goal was reached. This preserves the idea in expectation, but it can introduce high-variance targets, especially when rewards are small or the discount factor is close to one.

The human scientist proposed a soft alternative. Instead of sampling terminal success events, train the artificial success-goal head with expected terminal mass:
\[
    y(s,a,g_+) = (1-\gamma)\bar r(s,a) + \gamma \max_{a'} M_{\text{target}}(s',a',g_+),
\]
where \(g_+\) is an artificial success goal and \(M\) is a successor-measure-like value. The core hypothesis was that this soft target would match normalized Q-learning in tabular sanity checks while avoiding sampled terminal-event variance. A secondary hypothesis, intentionally separated from the first, was that auxiliary real-state successor goals might improve low-capacity function approximation through shared representations.

This project is easy to understand as an estimator question. Does replacing rare sampled success events with their expected mass preserve the desired value while reducing variance? The project is scientifically useful only if it avoids common pitfalls: inconsistent reward normalization, wrong terminal masks, policy comparisons without exact references, and untested auxiliary-goal claims. The charter therefore required exact or analytically known ground truth wherever possible and prohibited large neural experiments before tabular diagnostics passed.

\subsection{How the \intern\ approached the problem}

The \intern\ turned the idea into a ladder of experiments. The first experiment was the smallest possible diagnostic: a one-state MDP comparing sampled Bernoulli terminal targets with deterministic soft targets over several reward and discount values. This checked whether the two estimators have the same mean and different variance. A failure here would have immediately invalidated the estimator story.

The next stage tested tabular equivalence to normalized Q-learning. CliffWalking and small chain environments were used to verify scaling, terminal handling, and reward normalization. This was important because reward-to-goal conversions can appear successful while actually comparing values with inconsistent units. By using exact references, the system made the experiment a diagnostic rather than a benchmark race.

The middle stage tested propagation under coverage variation. RiverSwim-style chains examined whether soft updates help when reward propagation is long-horizon and whether any advantage depends on adequate coverage. The system separated adequate-coverage and starved-coverage regimes, which later prevented overclaiming. The result was not ``soft always wins''; it was ``soft helps under adequate coverage, but coverage-starved settings remain caveated.''

A later stage tested vector successor-measure correctness in a tiny FourRooms setting. Here the system checked that independent real-state goal slices and the artificial success-goal slice could be represented correctly. This was still a correctness check, not a shared-representation claim. Only after this did the system test a low-capacity shared low-rank model in which real-state auxiliary goals and the artificial reward head interacted through shared parameters.

Finally, when the low-rank auxiliary experiment produced negative transfer, the \intern\ did not simply search for a better framing. It ran a predeclared repair diagnostic: terminal-only reproduction, combined auxiliary reproduction, loss-balanced combined training, and staged real-goal pretraining followed by artificial-goal fine-tuning. These repairs also failed to match terminal-only performance. The system then moved to synthesis rather than more sweeping.

\subsection{Results and decisions}

The final project state is nuanced. The positive estimator evidence is strong within the intended small-tabular scope. In the one-state diagnostic, sampled and soft targets matched in mean, while the soft target had zero terminal-sampling variance and the sampled target variance reached approximately 0.0904. Scaling checks were near numerical precision: finite-MDP scaled error around \(3.95\times 10^{-8}\), CliffWalking scaled error around \(9.71\times 10^{-10}\), and FourRooms vector \(g_+\) scaled-vs-Q error around \(1.11\times 10^{-16}\). RiverSwim results supported Bellman/value improvements only under adequate coverage; starved-coverage runs remained limitations.

The auxiliary result was negative and carefully scoped. In the low-rank FourRooms experiment, terminal-only training had mean scaled value error around 0.073 and Bellman residual around 0.00096. Combined auxiliary training worsened value error to approximately 16.89 and Bellman residual to approximately 0.0364. Repair variants did not recover terminal-only performance. The \intern\ recorded the strongest defensible auxiliary claim as: the tested rank-4 low-rank FourRooms real-state auxiliary approach is unsupported and showed negative transfer. It also explicitly rejected broader claims such as ``auxiliary goals are generally harmful'' or ``soft reward-to-goal conversion works in neural benchmark settings.''

The final synthesis step produced an internal short-paper draft from existing evidence only. No new learning compute was run. The draft included a claim-to-evidence map, limitations, unsupported-claims red lines, and a pre-publication checklist. This is the intended behavior of the \intern: it accelerates the transition from rough idea to scoped evidence, but it does not decide that the result is externally publishable without human review.

\section{Discussion}

The case studies suggest that the \intern\ can accelerate scientific work in ways that are not captured by benchmark accuracy alone. The most obvious benefit is implementation throughput: the system writes scripts, runs diagnostics, and maintains artifacts. But the deeper benefit is epistemic structure. It forces the question ``what claim did this experiment test?'' and requires the answer to be linked to metrics, baselines, and artifacts.

This matters because many scientific projects fail not through one dramatic error but through gradual overclaiming. A method works in one small setting, then the claim expands; a baseline is missing; a failure mode is noted but not integrated; an agent proposes another sweep because it can. The \intern\ counters this with gates. The local reviewer can pass, weak-pass, fail, or require human input. The checkpoint policy can escalate to \gptpro. The metric ledger can preserve negative signals. The progress summary can state that the goal is not solved. These mechanisms make it easier for a human scientist to say: continue, stop, or reframe.

The Stochastic Long-Horizon Goal Reaching project illustrates the value of stopping. The project produced positive diagnostics: raw transitive value learning can overestimate lucky stochastic paths; log-space transitive updates improve long-horizon propagation relative to Monte Carlo-only labels in censored settings; history/context helps under partial observability. However, fairer model-DP baselines explained the apparent algorithmic gains. A fully autonomous system optimized for producing novelty might continue searching for a positive framing. The \intern\ instead produced a stop decision: the evidence did not support a distinct stochastic-TRL algorithmic advantage. That negative boundary is scientifically useful.

The Soft Reward-to-Goal Conversion project illustrates a different value: claim splitting. The system supported a scoped estimator claim while rejecting a broader auxiliary-benefit claim. This is more useful than either blanket optimism or blanket pessimism. It tells the scientist what can be written, what cannot be written, and what would require a new hypothesis.

\section{Limitations}

This work has several limitations. First, we report live case studies rather than a controlled productivity evaluation. We do not yet measure human time saved, total token cost, number of agent calls per accepted result, or comparison against a manual baseline. These measurements are important future work.

Second, both projects are computational reinforcement-learning prototypes. They are well suited to the \intern\ because they can be decomposed into small scripts, exact-DP checks, tabular diagnostics, and artifact validation. Other scientific domains may require laboratory interfaces, simulators, literature search, human-subject constraints, robotic safety constraints, or domain-specific validation.

Third, the \intern\ relies on LLM agents that can still fail. They can write invalid JSON, time out, misunderstand a charter, or propose low-value experiments. The point of the system is not that agents become infallible, but that failures become structured and recoverable. A timeout is a result; a parse failure is a blocker; a reviewer disagreement is a checkpoint.

Fourth, \gptpro\ checkpoint supervision is not a substitute for expert scientific judgment. In our workflow, \gptpro\ is useful as a senior reviewer that can read compact packets and challenge local decisions, but it does not own the research agenda. Human review remains necessary before external publication, large-scale experiments, or claims that go beyond the evidence.

Finally, the current implementation is optimized for early-stage prototyping, not final scientific validation. It intentionally blocks large training runs, broad hyperparameter sweeps, and benchmark-scale experiments unless explicitly approved. This is a feature for the intern paradigm but a limitation for end-to-end autonomous discovery.

\section{Conclusion}

We introduced the \intern, a repository-centered pipeline for evidence-gated human--AI scientific collaboration. The system operationalizes a bounded AI research intern: it translates rough ideas into small experiments, executes and reviews those experiments, maintains structured memory, escalates high-stakes decisions, and preserves both positive and negative evidence. Across two live reinforcement-learning research projects, the \intern\ produced scoped positive claims, negative boundary results, and stopping decisions that prevented unsupported scaling. We believe this middle-ground autonomy---more capable than passive coding assistance, more accountable than unconstrained AI-scientist automation---is a promising design pattern for language models in scientific discovery.

\section*{Ethics and Reproducibility Statement}

The \intern\ is designed to assist human scientists, not replace expert review. It can generate code and scientific prose, so misuse could amplify unsupported claims. The pipeline therefore emphasizes structured evidence, claim boundaries, and human/\gptpro\ checkpoints before publication. Repository artifacts, schemas, prompts, results, reviews, and progress summaries are intended to make the workflow auditable and reproducible.

\section{Appendix: Stochastic Long-Horizon Goal Reaching Case Study}
\label{app:stochastic-long-horizon}

The second project asked whether stochastic variants of transitive value learning could improve long-horizon offline goal reaching. In ordinary deterministic goal-reaching tasks, a transitive or divide-and-conquer value update can use intermediate waypoints to propagate long-horizon reachability. The human scientist's concern was that under stochastic dynamics, such updates may become overoptimistic: if an offline dataset contains lucky stochastic shortcut outcomes, a deterministic-style transitive backup may treat the shortcut as reliably good.

The charter therefore framed the project around three questions. First, can raw transitive value learning overestimate risky stochastic paths? Second, does a log-space version improve numerical stability and long-horizon propagation? Third, can stochastic calibration, successor-distance structure, posterior uncertainty, or history/context produce a distinct improvement over fair baselines?

The \intern\ began with exact-DP tabular diagnostics. Early experiments confirmed that raw transitive value learning could prefer lucky stochastic shortcuts and that log-space transitive learning helped long-horizon propagation relative to Monte Carlo-only labels in censored settings. These were useful positive diagnostics: they identified a real failure mode and a useful baseline behavior.

The next experiments tested whether successor-distance or stochastic-calibrated variants produced distinct gains. They did not. Successor-distance variants that improved metrics were numerically equivalent to log-space transitive learning under the audit tolerance. A hand-shaped conservative penalty reduced one lucky-only overestimation case, but the rule was narrow. A more generic count-based uncertainty replacement reduced Q-overestimation modestly but did not fix the policy-level lucky-only failure.

The project then pivoted toward posterior and partial-observation settings. Posterior-transitive variants were compared to prior-matched posterior mean model-DP baselines and were found equivalent or near-equivalent. In aliased POMDPs, history/context helped strongly when cues were sufficient, but history-model DP explained the gains. The final \gptpro\ decision stopped the loop: the project had generated valuable diagnostics, but no reviewed experiment supported a distinct stochastic transitive-RL algorithmic advantage over the appropriate baseline.

\section{Appendix: Example Agent Prompts and Schemas}

The implementation uses short role prompts plus structured schemas. The exact wording can change, but the important feature is that each role has a narrow contract.

\subsection{Supervisor prompt sketch}

\begin{verbatim}
You are SUPERVISOR in an automated research loop.
You are a planner and decision-maker, not an executor.
Read the charter, state, latest results, and latest reviews.
Choose continue, pivot, stop, needs_human, or pro_review.
If continuing, propose one small experiment that can finish
within the configured timeout. Reward evidence, not activity.
Return JSON matching decision.schema.json.
\end{verbatim}

Required decision fields include \texttt{decision}, \texttt{confidence}, \texttt{progress\_score}, \texttt{rationale}, \texttt{evidence}, \texttt{risks}, and either a null or structured \texttt{next\_experiment}. The \texttt{next\_experiment} must specify an experiment id, objective, hypothesis, success criteria, failure criteria, tasks, required outputs, and estimated runtime.

\subsection{Executor prompt sketch}

\begin{verbatim}
You are EXECUTOR in an automated research loop.
Follow the supplied experiment plan.
Run only small-scale experiments suitable for quick validation.
Save exact commands, raw metrics, baseline metrics, artifacts,
a result JSON, and a summary Markdown file.
If the plan is impossible, write a failed result JSON explaining why.
Do not change the research goal or delete previous results.
\end{verbatim}

The result schema requires fields for \texttt{experiment\_id}, \texttt{status}, \texttt{claim\_tested}, \texttt{commands\_run}, \texttt{metrics}, \texttt{baseline\_metrics}, \texttt{artifacts}, \texttt{interpretation}, \texttt{known\_failures}, and \texttt{next\_questions}. Artifact paths are validated before the loop can continue.

\subsection{Reviewer prompt sketch}

\begin{verbatim}
You are REVIEWER in an automated research loop.
Audit the latest experiment. Check required files, artifacts,
baseline fairness, success criteria, failure criteria, cherry-picking,
overclaiming, and whether auto-continuation should be allowed.
Return JSON matching review.schema.json.
\end{verbatim}

The review schema requires a verdict, an auto-continuation flag, checked evidence, reasons, required fixes, and risk flags. This makes reviewer reflection inspectable rather than hidden in prose.

\subsection{Summarizer prompt sketch}

\begin{verbatim}
You are SUMMARY in an automated research loop.
Write a concise progress summary from reviewed evidence.
Separate confirmed findings from limitations and open questions.
Do not claim the research goal is solved unless charter success
criteria are satisfied by reviewed evidence.
\end{verbatim}

Summaries include current status, experiment ledger, main findings, limitations and risks, recommended next human decision, and files to inspect. They are used both for humans and as compact input to \gptpro\ checkpoints.

\subsection{GPT-5.5-Pro checkpoint prompt sketch}

\begin{verbatim}
You are the external GPT-5.5-Pro research supervisor.
Given the evidence packet, decide whether the project is making
real progress. Choose continue, pivot, stop, or needs_human.
Be skeptical. Reward evidence, not activity. If continuing, propose
one small experiment with success/failure criteria.
Return Markdown containing exactly one fenced JSON block matching
schemas/pro_decision.schema.json, followed by a short explanation.
\end{verbatim}

This role is intended as senior checkpoint supervision, preferably through an API-backed call in production. It is invoked selectively rather than at every local iteration.

\section{Appendix: Example Charter}

Below is a shortened charter for the Soft Reward-to-Goal Conversion project. It illustrates the level of human guidance expected before the agents begin.

\begin{verbatim}
Research goal:
Rapidly test whether a soft successor-measure reward-to-goal
conversion can preserve reward-to-goal equivalence while avoiding
sparse, high-variance sampled terminal success events.

Main hypothesis:
Training the artificial success-goal head with expected terminal mass,
  y(s,a,g+) = (1-gamma) r_bar(s,a) + gamma max_a' M_target(s',a',g+),
will match normalized Q-learning in tabular sanity checks while
removing terminal-sampling variance.

Initial scope:
Start with tiny discrete environments: one-state diagnostics,
CliffWalking, RiverSwim, and tiny FourRooms. Do not run MuJoCo,
AntMaze, OGBench, image observations, or expensive neural training.

Primary metrics:
Target mean/variance; number of sampled g+ events per 10k transitions;
max absolute value error between scaled M_plus and normalized Q;
policy disagreement; Bellman error; average return.

Failure criteria:
Pause if terminal-only soft successor learning fails to match normalized
Q-learning, if reward normalization is ambiguous, if terminal masks are
wrong, if only training loss is reported, or if larger benchmarks are
attempted before tabular diagnostics pass.
\end{verbatim}
